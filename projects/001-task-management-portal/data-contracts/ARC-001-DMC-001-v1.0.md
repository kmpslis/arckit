**Purpose**: The `task-data` data product provides authoritative, queryable access to all core task management entities within the Quento1 platform: tasks (including sub-tasks), projects, workspaces, comments, and a non-PII user profile projection. It is the single source of truth for task state and project progress, serving internal analytics, operational tooling, and future inter-domain consumers.

**Business Value**: This data product enables downstream consumers to build reporting dashboards, NPS analysis pipelines, billing analytics, and operational monitoring tooling without direct database coupling. It contributes directly to Outcome O-1 (£180K ARR) by enabling data-driven product iteration, and to Outcome O-3 (NPS 40+) by surfacing user behaviour insights to the Product Manager without requiring ad hoc database access. Aligned with Architecture Principle 8 (Single Source of Truth) and Principle 9 (Loose Coupling).

## 2. Schema (Data Structure)

### 2.1 Schema Overview

**Schema Version**: 1.0.0 (independent from contract version)

**Objects**: 5 objects defined
**Properties**: 59 total properties across all objects
**PII Fields**: 3 fields in USER_PROFILE object

**ODCS Terminology**:

- **Object** = Structure of data (table in RDBMS, document in NoSQL, file in data lake)
- **Property** = Attribute of an object (column, field, attribute)

> **Note**: All objects are exposed as read-only views over the Quento1 operational database. Consumers MUST NOT write to these objects directly. All writes go through the Quento1 REST API (INT-001 pattern).

### 2.2 Object Definitions

#### Object: TASK

**Description**: Core data product object. Represents a unit of work within a Project. Includes both top-level tasks and sub-tasks (distinguished by whether `parent_task_id` is null).

**Source Entity**: DR-003 (ARC-001-REQ-v1.1.md)

**Object Type**: TABLE

**Storage Location**: `quento1_prod.public.tasks` (operational database read replica)

**Access Pattern**: QUERY (SQL on read replica), API (REST endpoint)

**Properties:**

| Property | Type | Required | PII | Description | Business Rules |
|----------|------|----------|-----|-------------|----------------|
| id | UUID | Yes | No | Unique task identifier | Primary key, immutable |
| project_id | UUID | Yes | No | Parent project identifier | FK → PROJECT.id |
| workspace_id | UUID | Yes | No | Parent workspace identifier | FK → WORKSPACE.id |
| title | String(500) | Yes | No | Task title | Not null; max 500 chars |
| description | Text | No | No | Task body in Markdown | Nullable; max 50,000 chars |
| status | Enum | Yes | No | Current workflow status | to_do \| in_progress \| in_review \| done \| archived |
| priority | Enum | Yes | No | Importance level | critical \| high \| medium \| low |
| assignee_id | UUID | No | No | Assigned user (anonymised reference to USER_PROFILE.id) | Nullable; FK → USER_PROFILE.id |
| created_by | UUID | Yes | No | Creator user (anonymised reference) | FK → USER_PROFILE.id |
| due_date | Date | No | No | Target completion date | Nullable; ISO 8601 |
| parent_task_id | UUID | No | No | Parent task for sub-tasks (NULL = top-level task) | Nullable; FK → TASK.id; max nesting depth = 1 |
| created_at | Timestamptz | Yes | No | Creation timestamp | ISO 8601 UTC; immutable |
| updated_at | Timestamptz | Yes | No | Last modification timestamp | Updated on any field change |
| archived_at | Timestamptz | No | No | Archive timestamp | Null = active; set on archive action |
| deleted_at | Timestamptz | No | No | Soft delete timestamp | Null = not deleted; excluded from default queries |

**Primary Keys**: `id`

**Foreign Keys**: `project_id → PROJECT.id`, `workspace_id → WORKSPACE.id`, `assignee_id → USER_PROFILE.id`, `created_by → USER_PROFILE.id`, `parent_task_id → TASK.id`

**Indexes**: `(project_id, status)`, `(assignee_id, status)`, `(workspace_id, due_date)`, `(parent_task_id)` WHERE parent_task_id IS NOT NULL, PARTIAL on `(project_id)` WHERE deleted_at IS NULL

**Partitioning**: Not partitioned at v1.0. Review at 500K records. Future: partition by `created_at` month.

**Default Filter**: Consumers SHOULD apply `WHERE deleted_at IS NULL` unless specifically querying deleted records (requires Admin role).

#### Object: USER_PROFILE

**Description**: A non-PII projection of the User entity, exposing only display-level information suitable for task attribution, assignment display, and analytics. Sensitive authentication fields (password_hash, mfa_secret) and direct PII requiring additional authorisation (email) are access-controlled.

**Source Entity**: DR-001 (ARC-001-REQ-v1.1.md)

**Object Type**: VIEW (database view over the users table; not a raw table exposure)

**Storage Location**: `quento1_prod.public.user_profiles_v` (database view)

**Access Pattern**: QUERY (SQL), API (REST)

**Properties:**

| Property | Type | Required | PII | Description | Business Rules |
|----------|------|----------|-----|-------------|----------------|
| id | UUID | Yes | No | User identifier | Primary key; FK target for TASK.assignee_id, TASK.created_by, COMMENT.author_id |
| display_name | String(100) | Yes | **Yes** | User's chosen display name | Not null; may contain real name |
| avatar_url | String(2048) | No | No | Profile picture URL | Nullable |
| created_at | Timestamptz | Yes | No | Account creation timestamp | ISO 8601 UTC |
| email | String(320) | Yes | **Yes** | User email address | **Restricted — Consumer-PII role required** |
| workspace_role | Enum | Yes | No | Role within queried workspace context | admin \| member \| viewer |
| deleted_at | Timestamptz | No | No | Soft delete timestamp | Null = active |

**Primary Keys**: `id`

**Indexes**: `(id)`, `(email)` — email index only available to Consumer-PII role

**PII Access Restrictions**:

- `email`: Requires **Consumer-PII** role (separate approval by DPO). Standard Consumer-Read role receives this field as `null`.
- `display_name`: Accessible to all Consumer-Read holders; treated as PII under UK GDPR as it may contain real name.
- Authentication fields (`password_hash`, `mfa_secret`, `mfa_enabled`, `email_verified`, `notification_prefs`) are **not exposed** in this data product.

### 5.2 API Specifications

**API Version**: v1.0.0

**Base URL**: `https://api.quento1.io/v1/data-products/task-data`

**Authentication**: OAuth 2.0 (client credentials flow) — aligned with Architecture Principle 4 (Security by Design)

**Rate Limits**:

- **Consumer-Read Tier** (internal): 300 requests/minute per consumer
- **Consumer-PII Tier** (restricted): 100 requests/minute; all requests logged
- **Burst Limit**: 2x sustained rate for up to 60 seconds; then throttled to sustained rate

**API Endpoints**:

```text
GET /objects/tasks
  - Retrieve tasks (paginated, max 100 per page)
  - Query params: limit, offset, project_id, workspace_id, assignee_id,
                  status, priority, due_before, due_after, parent_task_id,
                  include_deleted (requires Admin role)
  - Default filter: deleted_at IS NULL

GET /objects/tasks/{task_id}
  - Retrieve single task by ID
  - Includes sub_tasks[] array with count and completion status

GET /objects/projects
  - Retrieve projects (paginated)
  - Query params: limit, offset, workspace_id, include_archived

GET /objects/projects/{project_id}
  - Retrieve single project with task_count summary

GET /objects/workspaces/{workspace_id}
  - Retrieve workspace (scoped to consumer's authorised workspaces)

GET /objects/comments
  - Retrieve comments (paginated)
  - Query params: limit, offset, task_id, author_id

GET /objects/user_profiles/{user_id}
  - Retrieve user profile
  - email field is null for Consumer-Read; populated for Consumer-PII

GET /metadata
  - Returns contract metadata: version, schema_version, status, SLA summary

GET /quality
```

## 6. Security and Privacy

### 6.1 Data Classification

```python
import requests
import time

BASE_URL = "https://dev-api.quento1.io/v1/data-products/task-data"
HEADERS = {"Authorization": f"Bearer {oauth_token}"}

# Test 1: Schema Validation — task object has required fields
def test_task_schema():
    r = requests.get(f"{BASE_URL}/objects/tasks?limit=1", headers=HEADERS)
    assert r.status_code == 200
    task = r.json()["data"][0]
    required = ["id", "project_id", "workspace_id", "title", "status", "priority",
                "created_by", "created_at", "updated_at"]
    for field in required:
        assert field in task, f"Missing required field: {field}"

# Test 2: Deleted tasks excluded by default
def test_soft_delete_filter():
    r = requests.get(f"{BASE_URL}/objects/tasks?limit=100", headers=HEADERS)
    tasks = r.json()["data"]
    assert all(t["deleted_at"] is None for t in tasks), "Deleted tasks should not appear"

# Test 3: SLA — Response time within p95 target
def test_latency_sla():
    start = time.time()
    r = requests.get(f"{BASE_URL}/objects/tasks?limit=50", headers=HEADERS)
    latency_ms = (time.time() - start) * 1000
    assert r.status_code == 200
    assert latency_ms < 500, f"p95 target breached: {latency_ms:.0f}ms"

# Test 4: Authentication enforced
def test_auth_required():
    r = requests.get(f"{BASE_URL}/objects/tasks")  # No auth header
    assert r.status_code == 401, "Unauthenticated requests must be rejected"

# Test 5: PII field restricted for Consumer-Read role
def test_pii_restriction():
    r = requests.get(f"{BASE_URL}/objects/user_profiles/any-user-id", headers=HEADERS)
    profile = r.json()["data"]
    assert profile.get("email") is None, "email must be null for Consumer-Read role"

# Test 6: Nesting depth — sub-tasks returned correctly
def test_subtask_structure():
    r = requests.get(f"{BASE_URL}/objects/tasks?parent_task_id=null&limit=10", headers=HEADERS)
    for task in r.json()["data"]:
        assert task.get("parent_task_id") is None, "Query filter did not apply"

# Test 7: Rate limiting enforced
def test_rate_limit():
    responses = [requests.get(f"{BASE_URL}/objects/tasks?limit=1", headers=HEADERS)
                 for _ in range(310)]  # Exceed 300/min limit
    status_codes = [r.status_code for r in responses]
    assert 429 in status_codes, "Rate limiting should have fired at 300+ req/min"
```

## 13. Known Limitations and Issues

**Current Limitations** (v1.0):

1. **No Data Lake export**: Parquet/S3 batch access not available in v1.0; planned for v2.0 (see Section 14)

5. **Historical snapshots**: Data product reflects current state only; point-in-time historical queries require use of the activity log (not yet exposed as a data product object in v1.0)
